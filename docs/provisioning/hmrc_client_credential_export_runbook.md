# HMRC Client Credential Export Runbook

This runbook governs the `pc_0038` bridge from HMRC Developer Hub application setup into governed client-credential lineage.

## Scope

- read the HMRC sandbox application identity from the Developer Hub credentials surface
- export the HMRC `client_id` into governed metadata refs
- export the currently active HMRC client secret into governed secret refs
- bind the resulting credential record to:
  - callback profiles
  - scope truth
  - fraud-header profiles
  - token-binding profiles
  - authority provider-profile rows
- emit only sanitized repo-tracked evidence and lineage metadata

Out of scope:

- product-runtime sign-in or consent journeys
- raw token exchange against live HMRC endpoints
- production HMRC credential export
- secrets-manager or KMS/HSM bootstrap beyond the predeclared template law from `pc_0033`

## Public HMRC Guidance Revalidated On 2026-04-18

- HMRC issues the `client_id` when an application is added to the Developer Hub.
- HMRC states client secrets are generated by the developer and are equivalent to passwords.
- HMRC says client secrets should not be stored in plain text.
- HMRC says rotation is additive first: generate a new secret, move dependents to it, verify, then delete the inactive secret.
- HMRC currently allows up to `5` active client secrets per application.
- HMRC user-restricted flows forbid placing `client_secret` on the `/oauth/authorize` request and require the configured `redirect_uri` to be reused at `/oauth/token`.

## Canonical Taxat Doctrine

1. The sanitized sandbox application record from `pc_0035` remains the only repo-safe portal identity baseline.
2. The current sandbox OAuth profile from `pc_0036` remains the source of truth for callback refs and scope truth.
3. The fraud-profile evidence from `pc_0037` remains the source of truth for current interactive fraud-header refs.
4. The authority provider profile catalog remains the source of truth for environment-specific token-binding profile refs.
5. Raw HMRC client secrets stay in process memory only long enough to hand them to the governed vault writer.
6. Repo-tracked outputs may persist only:
   - aliases
   - fingerprints
   - vault refs
   - attestation refs
   - write receipts
   - binding refs
   - sanitized evidence refs

## Export Flow

1. Open the canonical sandbox application console.
2. Open `Manage application credentials`.
3. Read the application ID and client ID from the HMRC credentials page.
4. Cross-check callback, scope, fraud-profile, and token-binding refs against:
   - `config/authority/hmrc/oauth/hmrc_sandbox_oauth_profile.json`
   - `data/provisioning/hmrc_sandbox_profile_binding_evidence.template.json`
   - `data/analysis/authority_provider_profile_catalog.json`
5. If governed client-secret lineage already exists, adopt it and revalidate the binding rows instead of generating a duplicate secret.
6. If governed lineage does not yet exist:
   - open a manual checkpoint unless the run explicitly allows one-time secret reveal
   - reveal or generate the HMRC client secret only when the vault writer is ready
   - stream the revealed value directly into governed vault refs
   - discard the raw value from process memory
7. Persist:
   - `hmrc_client_application_inventory`
   - `hmrc_client_vault_binding`
   - `hmrc_client_secret_lineage`
   - one sanitized evidence manifest

## Binding Doctrine

The export is admissible only if it binds the credential set to all of the following:

- sandbox web callback `cb_sandbox_web`
- sandbox desktop callback `cb_sandbox_desktop`
- pre-production web callback `cb_preprod_web`
- pre-production desktop callback `cb_preprod_desktop`
- scope set `hmrc_income_tax_mtd_current_slice_interactive`
- fraud-header refs:
  - `fph_web_app_via_server`
  - `fph_desktop_app_via_server`
- token-binding refs:
  - `tb_sandbox_web_gateway_bound`
  - `tb_sandbox_desktop_gateway_bound`
  - `tb_preprod_web_gateway_bound`
  - `tb_preprod_desktop_gateway_bound`

Any mismatch between the current portal app and those frozen refs is a hard block. The export SHALL fail closed rather than silently rebinding a client secret into the wrong environment or authority posture.

## Rotation Doctrine

Taxat's stricter client-secret overlap rule remains:

- HMRC provider hard limit: `5`
- Taxat maximum concurrent overlap during controlled rotation: `2`

Safe sequence:

1. generate a new HMRC client secret
2. export it directly into governed vault refs
3. update dependent controlled runtimes to reference the new version
4. verify token exchange within the same binding lineage
5. attest the cutover
6. retire the inactive prior version

Unsafe operations:

- writing a revealed secret to repo files, `.env`, screenshots, traces, or shell history
- generating a new secret without recording its binding lineage
- retiring an older secret before attested cutover
- reusing one sandbox credential record for a production namespace

## Manual Checkpoint Rule

If the HMRC portal shows a one-time secret reveal and the run is not explicitly configured to ingest it immediately, the flow SHALL:

- open a manual checkpoint before the reveal
- suppress screenshots, DOM snapshots, and traces for that step
- persist only placeholder lineage and checkpoint refs
- resume only once the vault writer and evidence posture are confirmed ready

## Sanitized Evidence

Admissible evidence may include:

- masked portal-step notes
- safe DOM fingerprints
- vault write receipts
- attestation refs
- checkpoint refs

Evidence must never include:

- raw `client_id`
- raw `client_secret`
- raw OAuth codes
- cookies
- browser storage-state blobs
- copied portal secrets in markdown or JSON

## Validation Expectations
