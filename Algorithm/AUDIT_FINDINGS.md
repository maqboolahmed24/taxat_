# Forensic Audit Findings

This repository is a specification-first algorithm package. The defects below were identified during
forensic review; they are contract, schema, state-machine, determinism, control-flow, and
operator-experience defects that would produce incorrect or unsafe runtime implementations if left
unresolved.

## Current status

All numbered findings in this ledger are historical material defects that are now closed by the
current blueprint corpus. `AUDIT_FINDINGS.md` remains authoritative for what was found,
`contract_integrity_requirements.md` remains authoritative for the standing ambiguity that must stay
closed, and `PATCH_RESOLUTION_INDEX.md` remains authoritative for the concrete repair mechanisms
that keep the defects from drifting back.

No material finding may remain unowned or open at the current blueprint stage. Any future
forensic-grade defect SHALL either be added here with an explicit closure path or block later
corpus and acceptance closure until that path exists.

1. **Amendment scope lacked a required year-end basis.** Root cause: `requested_scope` allowed `amendment_intent` / `amendment_submit` without `year_end`. Impact: amendment flows could be instantiated against an invalid reporting basis.
2. **Submit scope was not chained to preparation scope.** Root cause: `submit` could appear without `prepare_submission`. Impact: live filing could skip packet-preparation invariants and preflight checks.
3. **Analysis mode could still request filing-oriented scopes.** Root cause: `mode = ANALYSIS` was not contractually barred from `prepare_submission`, `submit`, or amendment-transmit scopes. Impact: counterfactual runs could be mistaken for live-ready runs.
4. **Replay classification was not bound to replay runs.** Root cause: `replay_class` could be present on non-`REPLAY` manifests. Impact: audit lineage and live execution semantics could blur.
5. **Replay runs could still carry live transmit scopes.** Root cause: replay manifests were not forbidden from `submit` or `amendment_submit`. Impact: a replay implementation could accidentally mutate an authority.
6. **Provider environment references were optional even when live authority progression was requested.** Root cause: `provider_environment_refs[]` had no conditional minimum size. Impact: live or amendment-capable manifests could be frozen without the target environment.
7. **Continuation config inheritance semantics were underspecified.** Root cause: `config_inheritance_mode` existed only as prose. Impact: child-manifest config lineage could drift silently across retries or replays.
8. **Top-level manifest lineage could diverge from `continuation_set`.** Root cause: the model did not require the projection fields to mirror nested lineage values. Impact: two incompatible ancestry views could exist for the same manifest.
9. **Gate decisions lacked stable identifiers.** Root cause: `gateDecision` had no `gate_decision_id`. Impact: gate outcomes could not be safely deduplicated or audited.
10. **Gate decisions were not bound back to the manifest.** Root cause: `gateDecision.manifest_id` was absent. Impact: detached gate records could be replayed or misattributed across runs.
11. **Gate code was nullable.** Root cause: `gate_code` accepted null. Impact: downstream policy analytics could lose the exact blocking or review source.
12. **Gate severity was nullable.** Root cause: severity was optional in practice. Impact: workflow prioritization and observability would degrade under partially populated gate records.
13. **Gate overrideability was nullable.** Root cause: override posture was not a required invariant. Impact: a block could be impossible to distinguish from a review or pass-with-notice in downstream systems.
14. **Gate policy version was nullable.** Root cause: `policy_version_ref` was not mandatory. Impact: operators could not prove which frozen policy made the decision.
15. **Gate timestamp was nullable.** Root cause: `decided_at` was not required. Impact: ordering, replayability, and incident reconstruction would become unreliable.
16. **Gate reason codes could be empty.** Root cause: `reason_codes[]` defaulted to empty. Impact: hard blocks and manual review paths could be emitted without an explainable cause.
17. **Review and block gate outcomes did not require next actions.** Root cause: `next_action_codes[]` stayed optional even for `MANUAL_REVIEW`, `OVERRIDABLE_BLOCK`, and `HARD_BLOCK`. Impact: remediations would be non-deterministic and UI guidance incomplete.
18. **Hard blocks did not force non-overrideable semantics.** Root cause: `HARD_BLOCK` allowed any overrideability value. Impact: clients could surface illegal override affordances.
19. **Overridable blocks did not require an explicit override scope.** Root cause: `required_override_scope` was optional even when the decision required scoped override. Impact: approval systems would not know what authority to request.
20. **TrustSummary lacked a canonical schema identity.** Root cause: no `$id` was declared. Impact: schema bundle resolution and validator caching could become ambiguous.
21. **TrustSummary omitted artifact self-description.** Root cause: `artifact_type` and `contract` were absent. Impact: the artifact could not be verified against the frozen schema bundle.
22. **TrustSummary was open to undeclared fields.** Root cause: missing `additionalProperties: false`. Impact: ungoverned fields could creep into a supposedly deterministic artifact.
23. **TrustSummary missed execution-context fields.** Root cause: `execution_mode`, `analysis_only`, `non_compliance_config_refs[]`, and `counterfactual_basis` were not modeled. Impact: analysis-mode trust summaries could be mistaken for compliance outputs.
24. **Snapshot lifecycle was not schema-enforced.** Root cause: `lifecycle_state` was absent from the snapshot contract despite the state-machine dependency. Impact: consumers could accept invalid or transient snapshots as final.
25. **Snapshot also missed execution-context fields.** Root cause: schema lagged behind the common derived-artifact model. Impact: modeled or non-live snapshots could contaminate compliance decisions.
26. **DecisionBundle lacked artifact identity and contract metadata.** Root cause: no `artifact_type` or `contract` fields. Impact: persisted terminal outcomes could not be integrity-checked.
27. **DecisionBundle omitted execution-context fields.** Root cause: terminal artifacts were not marked as compliance or analysis outputs. Impact: downstream orchestration could confuse modeled and live pathways.
28. **DecisionBundle had no stable bundle identifier.** Root cause: only `manifest_id` was present. Impact: idempotent reloads and downstream references had to overload the manifest key.
29. **DecisionBundle lacked a compact top-level reason summary.** Root cause: only lower-level gate chains encoded reasons. Impact: every consumer would need to replay all gates to understand why the run ended.
30. **The sample decision bundle no longer matched the contract.** Root cause: examples were stale. Impact: integrators would implement against an invalid artifact shape.
31. **CandidateFact schema omitted execution-context controls.** Root cause: common derived-artifact fields were missing. Impact: provisional analysis facts could be confused with compliance facts.
32. **Promoted candidate facts could exist without supporting evidence.** Root cause: `supporting_evidence_refs[]` had no conditional minimum for `PROVISIONAL` or `CANONICAL`. Impact: fact promotion could bypass provenance.
33. **CanonicalFact schema omitted execution-context controls.** Root cause: contract lagged behind the shared artifact model. Impact: analysis-derived canonical facts could leak into compliance compute.
34. **Canonical facts could be canonicalized without supporting evidence.** Root cause: no evidence minimum when `promotion_state = CANONICAL`. Impact: authoritative facts could exist without provenance.
35. **Canonical facts could be canonicalized without source strength.** Root cause: `source_strength_tier` remained nullable even for canonical state. Impact: trust and parity scoring would lack evidence-quality input.
36. **Canonical facts could be canonicalized without freshness state.** Root cause: `freshness_state` stayed nullable for canonical records. Impact: stale data could masquerade as current during compute or filing.
37. **Evidence items could be emitted without lineage.** Root cause: `lineage_refs[]` was optional. Impact: extracted evidence could not be traced back to source transformations.
38. **Evidence items could be emitted without retention tags.** Root cause: `retention_tag` was nullable. Impact: erasure, legal-hold, and retention handling would become unenforceable.
39. **Conflict records could reference fewer than two facts.** Root cause: `involved_fact_refs[]` had no minimum cardinality. Impact: single-fact “conflicts” could pollute the reconciliation pipeline.
40. **Conflict records could omit reason codes.** Root cause: `reason_codes[]` had no minimum cardinality. Impact: blocking conflicts would not explain why promotion was halted.
41. **Data-quality scoring could divide by zero on invalid error budgets.** Root cause: `error_budget_d` was used directly in the validation formula. Impact: validation scoring could explode or become undefined.
42. **Data-quality aggregation could divide by zero on empty/invalid weights.** Root cause: `Σw` was assumed positive. Impact: completeness and quality scores could become NaN and poison trust synthesis.
43. **Compute formulas referenced raw requested scope instead of authorized runtime scope.** Root cause: reporting scope was derived from `requested_scope[]`. Impact: access masking or scope reduction could be bypassed in compute.
44. **Analysis-mode fact eligibility was undefined.** Root cause: the formulas only stated canonical facts without analysis-specific policy. Impact: provisional facts might be used inconsistently across implementations.
45. **Annual adjusted totals were referenced before being defined.** Root cause: `RuleEvaluation(...)` consumed `annual_adjusted_totals` without a deterministic construction rule. Impact: engines could invent incompatible intermediate structures.
46. **Forecast randomness could vary by manifest identity.** Root cause: Monte Carlo seeding allowed `manifest_id`. Impact: identical frozen inputs could yield different forecasts after recovery or replay.
47. **Parity comparison sets were not canonicalized.** Root cause: `K` had no dedupe or deterministic ordering rule. Impact: parity scores could vary across runtimes or collection order.
48. **Parity thresholds and weights were not validated.** Root cause: weights, thresholds, and floors allowed invalid values. Impact: division-by-zero, negative pressure, or silent threshold disablement could occur.
49. **Relative parity delta could divide by zero on small values.** Root cause: denominator used `abs_floor_k` without enforcing positivity or minimum relative floor. Impact: zero-ish comparisons could generate infinite breach ratios.
50. **Parity math could propagate NaN or infinite numeric inputs.** Root cause: there was no invalid-input path for non-finite values. Impact: a single bad field could corrupt aggregate parity results.
51. **Per-field classification labels were inconsistent with aggregate logic.** Root cause: field classes used `MINOR`, `MATERIAL`, and `BLOCKING` while aggregate rules referenced `*_DIFFERENCE`. Impact: downstream classification logic could mis-map field states.
52. **Aggregate parity reused an ambiguous weight sum symbol.** Root cause: `Σw` was overloaded from earlier sections. Impact: readers and implementations could use the wrong denominator.
53. **Mandatory parity with zero comparable items lacked an explicit failure posture.** Root cause: no rule covered the `Σw = 0` parity case. Impact: runs could falsely appear comparable.
54. **Trust could still go GREEN while live authority state was unresolved.** Root cause: trust synthesis ignored unresolved baseline submission state when live progression was requested. Impact: the system could mark a run submission-ready without authoritative reconciliation.
55. **Automation could still be ALLOWED during unresolved authority progression.** Root cause: automation caps did not account for `UNKNOWN` or `OUT_OF_BAND_UNRECONCILED` baseline states. Impact: live automation could proceed under legal uncertainty.
56. **Filing readiness could be READY_TO_SUBMIT without resolved authority state.** Root cause: readiness permitted `LIMITED` automation and ignored unresolved baseline state. Impact: operators could be told to submit when reconciliation posture was not safe.
57. **Reason-code vocabulary lacked data-quality and parity-invalid-set signals.** Root cause: `DQ_INVALID_ERROR_BUDGET`, `DQ_WEIGHT_PROFILE_INVALID`, and `PARITY_COMPARISON_SET_INVALID` were absent. Impact: hardened failures would have no stable machine-readable explanation.
58. **Amendment gate prose had ambiguous boolean precedence.** Root cause: `and`/`or` conditions were written as free text. Impact: two compliant implementations could make opposite decisions on the same inputs.
59. **Drift lifecycle naming diverged from amendment semantics.** Root cause: state-machine labels used `MATERIAL_CHANGE` / `AMENDMENT_RECOMMENDED` while other docs used different terms. Impact: UI, workflow, and policy surfaces could drift out of sync.
60. **Authority canonicalization was under-specified.** Root cause: request path, query, header ordering, and Unicode normalization were not frozen precisely. Impact: identical semantic requests could hash differently and break dedupe or audit replay.
61. **Replay safety was not explicit at authority preflight.** Root cause: preflight rules did not state that replay runs must never perform live mutation. Impact: a recovery/replay code path could accidentally transmit.
62. **Request hashing omitted critical namespace dimensions.** Root cause: `request_hash` and `idempotency_key` did not include authority/environment/operation-family scoping. Impact: unrelated requests could collide under the same hash or idempotency surface.
63. **Experience deltas lacked a machine-readable attention policy.** Root cause: dominant issue/action posture existed only as loose top-level convenience fields without ordered rationale or default detail-module selection. Impact: clients could rank issues differently, reintroduce competing primaries, and drift back into clutter.
64. **Low-noise cognitive budgets were prose-only.** Root cause: surface, reason, warning, secondary-action, and detail-entry limits were not frozen in a contract artifact. Impact: incremental UI changes could silently erode the calm-shell guarantee.
65. **Composite shell payloads were schema-open.** Root cause: `surface_updates[].payload` accepted arbitrary objects even for `CONTEXT_BAR`, `DECISION_SUMMARY`, `ACTION_STRIP`, and `DETAIL_DRAWER`. Impact: implementations could produce incompatible summary/action/drawer semantics while still passing envelope validation.
66. **The renderer was not required to omit non-legal or non-material actions.** Root cause: the contract bounded primary actions but did not explicitly forbid disabled-control clutter. Impact: users could still face noisy action strips full of unavailable affordances.
67. **Detail entry-point relevance and cap were under-specified.** Root cause: `detail_entry_points[]` had no ordered default-module rule or bounded cardinality. Impact: the expert drawer could become another cluttered navigation surface.
68. **Focus-anchor continuity was not first-class in the read contract.** Root cause: reconnect guidance preserved the shell generally but did not persist an explicit `focus_anchor_ref` or shell stability token. Impact: live refreshes could disorient users by shifting the object they were reading.
69. **Empty, limited, latent, and not-applicable states were not normalized.** Root cause: the contract described empty states in prose but did not classify them formally. Impact: users could not reliably tell whether data was absent, still materializing, policy-hidden, or irrelevant.
70. **The design skill still prescribed theatrical multi-surface defaults.** Root cause: `UIUX_DESIGN_SKILL.md` continued to favor prismatic, cinematic, multi-region observatory language even though `low_noise_experience_contract.md` defines a calm-shell production profile. Impact: design and frontend teams could reintroduce split attention, metaphor overhead, and unnecessary ornament.
71. **The low-noise profile did not constrain top-level surface emission to the four composite shell surfaces.** Root cause: `ExperienceDelta` and `experienceFrame` still accepted rich observatory surfaces as peer top-level updates under `experience_profile = LOW_NOISE`. Impact: clients could validate and render a cluttered shell while still appearing contract-compliant.
72. **Microcopy brevity was not machine-enforced.** Root cause: context, summary, action, and detail-entry labels had no bounded low-noise copy budget. Impact: verbose authority payloads or long reason chains could displace the primary action and turn the calm shell into a scrolling spec dump.
73. **Default-shell and investigation-mode boundaries remained under-specified.** Root cause: rich observatory modules were still described as if they could coexist as ordinary first-view regions rather than progressive-disclosure depth. Impact: teams could ship investigator complexity in first view, increasing learning cost and slowing expert throughput.
74. **Analysis mode still admitted amendment-intent scope.** Root cause: the mode guard excluded live filing and amendment transmit tokens but still allowed `amendment_intent`. Impact: non-live runs could enter amendment workflow posture and blur compliance boundaries.
75. **Access, gate, manifest, and input-freeze lineage arrays still admitted duplicates.** Root cause: many refs and reason/action arrays were modeled as unconstrained lists instead of canonical sets. Impact: hashes, audits, workflow rollups, and replay-safe lineage could drift under semantically identical inputs.
76. **The replay-safe `DecisionBundle` bridge was still only partially schema-required.** Root cause: outcome-class, truth/checkpoint posture, plain reason, action bridge, and supporting arrays were not all mandatory on the persisted artifact, and legacy next-action rules could contradict `NO_SAFE_ACTION`. Impact: terminal reload could still fall back to client inference or contradictory shell states.
77. **TrustSummary cross-field posture remained under-constrained.** Root cause: the schema did not fully encode the documented relationship between `automation_level`, `trust_band`, `trust_level`, `filing_readiness`, and `required_human_steps[]`. Impact: integrators could validate contradictory trust artifacts such as `ALLOWED` automation without a truly ready-to-submit posture.
78. **Source-strength tiers remained open across source, evidence, and fact artifacts.** Root cause: the tier ladder was still described as recommended prose rather than enforced as a closed contract vocabulary. Impact: parity pressure, trust synthesis, and source ordering could drift across implementations.
79. **Artifact set wrappers still permitted duplicate items.** Root cause: set-style envelopes such as `SourceRecordSet`, `EvidenceItemSet`, `CandidateFactSet`, `CanonicalFactSet`, and `ConflictSet` did not require unique `items[]`. Impact: `set_hash` values and replay artifacts could change under duplicated but semantically identical content.
80. **Low-noise deltas still allowed multiple patches for the same composite surface.** Root cause: `surface_updates[]` had a four-item cap but no per-surface uniqueness rule. Impact: a single delta could carry contradictory updates for the same mounted shell region and break stable reconnect-safe rendering.
