- which files are canonical algorithm sources vs prompt scaffold vs non-canonical residue
- validator entrypoints discovered and whether they executed successfully
- prompt-scaffold cards discovered and their slug / sequence metadata

- anomalies, assumptions, and risks

2. Treat `Algorithm/` as the canonical source corpus and `PROMPT/` as the execution scaffold.
3. Record all top-level folders and normalize every path to repo-relative form. Produce a path-normalization map that

5. Recompute file counts from the live tree. Do not trust earlier assumptions or comments.
6. Validate the README claim that every top-level markdown document in `Algorithm/` appears exactly once in
   `README.md -> What you get`. Fail the task if any canonical top-level markdown file is missing, duplicated,
   or described under the wrong role.
7. Run both authoritative validation commands and persist their pass/fail outcomes, stdout summaries, and timestamps
   in machine-readable form. If a validator fails, capture the exact failure and stop treating the corpus as intake-clean.
8. Build a source-precedence memo that future agents can consume. It must explicitly distinguish:
   - core algorithm sources,
   - canonical object/state/formula/protocol sources,
   - specialized bounded-domain contracts,
   - enforcement artifacts,
   - support/coherence docs,
   - historical closure docs.
9. Separate canonical source files from non-canonical archive residue such as `__MACOSX/` and `.DS_Store`.

Validation rules:
- Both validator entrypoints must execute successfully, or the repo cannot be marked intake-clean.
- Every canonical top-level markdown file must be represented exactly once in the README inventory.
- Every first-five prompt card must be present and match its checklist slug.
- Every canonical file must have a deterministic checksum and canonical-vs-residue classification.
- Every absolute path found in the prompt scaffold must have a repo-relative normalized equivalent in your mapping.

Definition of done:
- A new agent can open your intake artifacts and know exactly what the canonical source set is, how to validate it,
  which prompt cards are next, and which local-path assumptions have been normalized away.
- The intake evidence pack is deterministic, machine-readable, and sufficient to trust the repository as the starting point for tasks 002-005.
