Execution steps:
1. Parse `data_model.md` and classify every named object family into one of the canonical kinds above.
2. Map each object to its backing schema(s), if any.
3. Create a truth/projection/control boundary matrix that explicitly separates:
   - mutable operational entities,
   - append-only compliance artifacts,
   - control contracts and envelopes,
   - read models and viewer-safe projections,
   - external-authority mirrors.
4. Build a schema ownership matrix showing:
   - which schema belongs to which conceptual object,
   - whether it is canonical object storage, a control contract, a read model, a transport envelope, or a regression harness,
   - and which prose document is authoritative for its semantics.
5. Build a read-model projection index that distinguishes staff/operator, customer/client, governance, and authority-facing surfaces.

   - staff workflow/collaboration views vs customer-safe request/detail projections
   - internal authority state vs projected filing/trust status

- Close the “no lifecycle coverage audit per object family” gap.
- Close the “read-model visibility families are scattered across profile-specific docs” gap.

- Embedded control contracts inside larger artifacts must not be mistaken for top-level mutable entities unless the corpus explicitly promotes them.

- absence of a lifecycle state is itself meaningful; do not fabricate lifecycle machines where the corpus defines a static or append-only artifact instead.
