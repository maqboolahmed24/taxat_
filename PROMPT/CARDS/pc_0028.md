Default hypothesis to test rigorously:
- the winning layout is a domain-oriented monorepo with:
  app surfaces for operator web, client portal web, native macOS, and backend/control plane,
  plus a restrained set of shared packages for contracts, generated types, domain kernels,
  access/session, manifests/replay, authority gateway, workflow/collaboration, projectors/read models,
  frontend design system / selector grammar / route runtime, testing fixtures, telemetry, and release tooling.
If the evidence rejects that, say so and choose the stronger topology.

Execution steps:
1. Parse the later roadmap tracks and map each implementation task to the package or packages it most naturally belongs to.
2. Build a boundary matrix that states, for every package:
   responsibilities, source contracts, allowed dependencies, forbidden dependencies,
   generated-code boundaries, and likely future tasks that land there.
3. Build a team ownership map that groups packages into coherent streams rather than one team per folder.
4. Compare the required topology alternatives with weighted criteria and explicit rejection reasons.
5. Draft CODEOWNERS-style ownership suggestions that align with the package map.
6. Specify the dependency rules:
   - which packages are leaf apps,
   - which are shared foundations,
   - which may depend on generated types,
   - which may not import UI code,
   - which may not depend on app shells,
   - and how test/support packages may legally span layers.
7. Fence deferred decisions such as exact package manager or workspace tooling details

- Close the “team ownership and review boundaries are absent” gap.
- Close the “schema / generated-type / contract ownership is easy to blur” gap.
- Close the “frontend shared packages versus app-specific shells are not formally separated” gap.
- Close the “later autonomous agents need package destinations before implementation begins” gap.

Mandatory edge cases:
- Generated schema/type outputs must have a single source-of-truth package boundary.
- UI packages must not become dumping grounds for backend legality or business rules.
- Native and web surfaces may share contracts and selector semantics, but not arbitrary runtime-specific UI code.
- Test harness packages must avoid duplicating domain semantics that should live in shared kernels or contract packages.
- Authority gateway code must stay isolated from browser packages and from read-side-only packages.
- Release and migration tooling must stay separate from feature UI packages.
- Avoid circular dependency chains and avoid micro-package sprawl that creates more indirection than clarity.
- Every later roadmap task from phases 02–06 must map cleanly to at least one owning package and one owning team.

Architecture, backend, frontend, native, and security requirements:
- Be explicit about allowed dependency directions and ownership handoffs.

Validation rules:
- At least three serious topology alternatives must be compared.
- Every later track cluster must map to packages and owners.
- The dependency rules must be acyclic or record any unavoidable exception as a typed risk.
- The package map must make later implementation destinations obvious enough that an autonomous agent can choose a package without re-arguing repo structure.
- Any unresolved ambiguity must be emitted as a typed gap or deferred decision.

Definition of done:

  what it may depend on, who owns it, and how the repo avoids both monolith sprawl and micro-package chaos.
