```text
Execute roadmap task `phase_00_seq_003_build_contract_schema_script_cross_reference_index`.

Inherit the shared operating contract and consume the outputs from tasks `pc_0001` and `pc_0002`.

Mission:
Build the authoritative cross-reference index that links prose contracts, schemas, sample payloads,
validator logic, forensic guard logic, and live constraint-traceability records into one machine-readable map.
This task is the bridge between “what the algorithm says” and “what the repository can actually enforce.”

Algorithm grounding you must honor:
- `data_model.md` defines the canonical object families that later schemas and validators are expected to enforce.
- `state_machines.md`, `invariants_and_gates.md`, `exact_gate_logic_and_decision_tables.md`,
  `compute_parity_and_trust_formulas.md`, `authority_interaction_protocol.md`, and other bounded contracts
  define the semantics that schemas and scripts must mirror.
- `README.md -> Reference document roles` says `constraint_traceability_register.json` is the machine-checkable live
  constraint register and that `validate_contracts.py` plus `forensic_contract_guard.py` validate the live traceability spine.
- `scripts/validate_contracts.py` constructs the schema registry, validates samples, and contains custom invariant checks
  that are not fully represented by JSON Schema alone.
- `tools/forensic_contract_guard.py` contains cross-document and cross-schema guardrails for high-risk integrity rules.
- `AUDIT_FINDINGS.md`, `PATCH_RESOLUTION_INDEX.md`, and `contract_integrity_requirements.md` describe why some of these
  enforcement mechanisms exist and which historical defects they keep closed.

Deliverables to create:
1. `docs/analysis/03_contract_schema_script_cross_reference_index.md`
2. `docs/analysis/03_enforcement_coverage_report.md`
3. `docs/analysis/03_contract_enforcement_graph.mmd`
4. `data/analysis/contract_schema_script_index.json`
5. `data/analysis/contract_schema_script_index.csv`
6. `data/analysis/object_family_to_enforcement_map.json`
7. `data/analysis/enforcement_gap_register.json`
8. `tools/analysis/build_cross_reference_index.py`
9. `tools/analysis/validate_cross_reference_index.py`

Each cross-reference row must capture at minimum:
- `logical_family_id`
- `logical_family_name`
- `family_kind` (`object_family`, `contract_family`, `state_machine_family`, `validator_family`, `forensic_guard_family`, `constraint_family`)
- `authoritative_prose_refs[]`
- `schema_refs[]`
- `sample_refs[]`
- `validator_refs[]`
- `forensic_guard_refs[]`
- `constraint_register_refs[]`
- `historical_closure_refs[]`
- `coverage_status` (`fully_mapped`, `partially_mapped`, `doc_only`, `schema_only`, `validator_only`, `gap`)
- `gap_notes[]`

Execution steps:
1. Start from `data_model.md` and extract the canonical object families, then expand outward into the contracts that serialize,
   constrain, or interpret each family.
2. Parse every schema in `schemas/` for `$id`, title, and obvious owning object/contract names.
3. Parse `scripts/validate_contracts.py` for:
   - schema-registry construction,
   - sample-loading conventions,
   - custom invariant functions/blocks,
   - issue kinds,
   - self-test pathways.
4. Parse `tools/forensic_contract_guard.py` for:
   - named guard themes,
   - document-heading dependencies,
   - schema/document/readme alignment checks,
   - traceability-register expectations.
5. Link every live constraint entry in `constraint_traceability_register.json` to:
   - authoritative prose refs,
   - enforcement refs,
   - downstream refs,
   - examples or vectors where available.
6. Mine `AUDIT_FINDINGS.md` and `PATCH_RESOLUTION_INDEX.md` to identify enforcement mechanisms that exist because of prior failures.
   Do not turn historical documents into primary semantics, but do use them to explain why enforcement coverage exists or is missing.
7. Build a graph that shows the chain:
   prose contract -> schema -> sample -> validator -> forensic guard -> constraint register -> historical closure context.
8. Classify enforcement coverage rigorously:
   - schema + custom validator + forensic guard
   - schema + validator only
   - prose + schema only
   - prose only
   - validator/guard without clear prose anchor
   - unresolved gap
9. Emit gaps explicitly whenever you find:
   - prose object families without schema or enforcement,
   - schemas with no owning prose contract,
   - custom validator logic with no clear contract anchor,
   - live constraint entries without real enforcement refs,
   - historical findings that appear closed but cannot be mapped to current mechanisms.

Gap handling you must perform:
- Close the “enforcement lives in multiple places with no single index” gap by building one canonical map.
- Close the “schema vs custom invariant vs forensic guard coverage is opaque” gap by classifying enforcement depth explicitly.
- Close the “historical closure mechanisms are not connected to live enforcement” gap by linking current guardrails to their closure context.
- Close the “object family ownership is discoverable only by reading many files” gap by creating an object-family-to-enforcement map.
- Close the “constraint register is machine-readable but not implementation-friendly” gap by mapping every live constraint to concrete artifacts.

Mandatory edge cases:
- Some constraints are enforced by document-term checks rather than schema fields; preserve that distinction.
- Some logical families may map to many schemas or many prose contracts; do not force false one-to-one simplifications.
- Helper code in `validate_contracts.py` that is generic infrastructure should not be mislabeled as domain semantics.
- Historical findings may be resolved by multiple mechanisms; preserve many-to-many relationships instead of flattening them.

Validation rules:
- Every live constraint in `constraint_traceability_register.json` must appear in the cross-reference index.
- Every schema file must map to at least one logical family or be explicitly classified as utility/supporting.
- Every custom invariant block and forensic guard theme must have at least one owning prose anchor or a documented gap.
- No row may silently mix “primary semantic owner” and “historical context only” references.
- The graph and tabular index must agree on the same identifiers.

Definition of done:
- A later backend, QA, or prompt-authoring agent can answer “where is this rule defined, what schema carries it,
  what code enforces it, and what historical defect made it necessary?” from one index instead of manual archaeology.
```
