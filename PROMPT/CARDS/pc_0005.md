Mission:
Convert the invention boundary and system boundary of the Taxat corpus into a rigorous, implementation-grade
requirement set. This is the task that prevents later roadmap phases from accidentally implementing
controlled-edge, authority-owned, or entirely external behavior inside the core engine. Your output must
make the inside/outside boundary queryable, testable, and hard to misread.

Algorithm grounding you must honor:
- `invention_and_system_boundary.md -> 1) Exact invention boundary` defines the invention’s start point,
  end point, positive boundary, inventive center, negative boundary, technical effect, and preferred embodiment.
- `invention_and_system_boundary.md -> 2) System boundary` defines the three-zone model:
  core engine zone, controlled edge zone, and external authority / external actor zone.
- `invention_and_system_boundary.md -> D. Boundary rules the spec should state explicitly`
  defines non-negotiable rules such as:
  - the engine owns decisions, not identity issuance,
  - the engine owns canonicalization, not raw-source truth,
  - the engine owns filing intent and packet formation, not authority acceptance,
  - the engine owns parity and trust, not HMRC’s own calculations,
  - the engine owns amendment recommendation, not unrestricted amendment rights,
  - the engine owns lineage-safe replay/recovery/continuation decisions,
  - the engine owns exact-scope posture,
  - the engine may guide users to HMRC online services but does not absorb HMRC-only tasks,
  - the engine owns evidence-linked explainability for the states it creates,
  - the engine owns compliance-vs-analysis segregation.
- `core_engine.md -> Summary (technical solution)` and `core_engine.md -> Boundary specification`
  anchor the boundary back to the end-to-end run sequence.
- `canonical_source_and_evidence_taxonomy.md` is critical for separating raw source truth from canonical engine truth.
- `actor_and_authority_model.md` and `authority_interaction_protocol.md -> 9.1 Boundary rule` / `9.2 Protocol scope`
  define actor, delegation, authority, and controlled-authority-transport boundaries.
- `northbound_api_and_session_contract.md`, `security_and_runtime_hardening_contract.md`,
  `deployment_and_resilience_contract.md`, `retention_and_privacy.md`, and `replay_and_reproducibility_contract.md`
  clarify what the engine owns semantically versus what surrounding infrastructure or external systems own operationally.

Deliverables to create:

Each boundary requirement row must capture at minimum:
- `boundary_requirement_id`
- `capability_name`
- `zone` (`inside_core_engine`, `controlled_edge`, `broader_product_outside_core`, `external_authority_or_actor`)

- `downstream_phase_implications[]`
- `notes`

Execution steps:
1. Extract every atomic capability listed or implied in `invention_and_system_boundary.md`, starting with the positive boundary,
   the system-boundary zones, and the explicit boundary rules.
2. Build a capability matrix that classifies each capability into one of four zones:
   - inside the core engine,
   - controlled edge that the engine depends on but does not invent,
   - broader product functionality outside the core engine but still relevant to the overall product,
   - entirely external authority or external-actor behavior.
3. Use `core_engine.md`, `data_model.md`, and bounded-domain contracts to attach each in-scope capability to the
   engine objects and artifacts that actually own it.
4. For every out-of-scope or adjacent capability, explain exactly why it is out-of-scope and what the engine
   is still allowed to do about it (for example: guide, integrate, validate, request, compare, record, or react).
5. Make the following distinctions explicit and testable:
   - engine canonicalization vs raw source truth
   - internal submission intent vs authority acceptance
   - parity/trust synthesis vs authority’s own calculation
   - amendment recommendation vs legal right to amend
   - core engine semantics vs northbound API transport
   - core engine semantics vs browser/native/client rendering
   - replay/recovery lineage governance vs generic infrastructure retry behavior
   - controlled connector use vs generic ETL / generic OCR / generic workflow claims
6. Produce a context diagram that shows the core engine, controlled-edge services, broader product surfaces,
   and external authority/external actors with clearly labeled responsibilities.
7. Derive downstream phase implications. Each row should help later tasks know whether they are implementing
   engine logic, integrating a controlled-edge dependency, or consuming external truth.
8. Include a dedicated out-of-scope-but-adjacent register for things the roadmap may touch operationally
   but must not absorb into the invention boundary.

Gap handling you must perform:
- Close the “boundary rules are prose only” gap by exporting atomic requirement rows and a capability matrix.
- Close the “inside vs controlled-edge vs external responsibilities are easy to blur” gap by classifying every major capability.
- Close the “out-of-scope does not mean irrelevant” gap by documenting adjacent external functions and the engine’s lawful relationship to them.
- Close the “later phases may accidentally build external-actor behavior into core logic” gap by producing downstream phase implications.
- Close the “boundary semantics are not queryable by code or prompts” gap by shipping machine-readable JSONL/CSV plus a validation tool.

Mandatory edge cases:
- HMRC sign-up, agent authorization setup, HMRC online-services-only tasks, authority-owned legal status,
  and HMRC’s own calculation service must not be pulled into the core-engine zone.
- UI styling, dashboards, and generic front-end layout are outside the invention boundary as such, even though
  engine-authored interaction semantics and route-visible posture are inside the semantic boundary.
- Controlled-edge fetching is not the same thing as raw external truth becoming canonical.
- Human judgment outside the approved override protocol must remain outside the engine’s deterministic core.
- Replay, recovery, continuation, and supersession lineage decisions are inside the governed engine boundary,
  not “just infrastructure.”
- Compliance-vs-analysis segregation is itself a boundary requirement and must not be treated as an implementation detail.

Architecture and backend rigor you must impose:
- Any later-facing interface or storage implication you describe must follow fail-closed boundaries,
  least privilege, immutable run/lineage semantics, deterministic hashing, idempotent command handling,
  explicit state transitions, durable audit evidence, typed errors, and secure partition isolation.
- Do not recommend shortcuts that would mix semantic engine responsibility with transport convenience,
  UI convenience, or external-system heuristics.

Validation rules:
- Every major inside-engine capability named in `invention_and_system_boundary.md` must appear in the matrix.
- Every major out-of-scope or adjacent capability named there must also appear in the matrix or out-of-scope register.
- Every explicit boundary rule from section D must map to at least one atomic requirement row.
- No capability may be classified `inside_core_engine` if the source says the engine only guides, integrates with,
  or reacts to that behavior.
- No out-of-scope row may omit the engine’s lawful relationship to that external function.

Definition of done:
- A later architect or coding agent can use your boundary pack to know exactly what Taxat should implement,
  what it should integrate with, what it should never absorb, and which invariants protect that separation.
