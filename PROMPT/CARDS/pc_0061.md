- which schema families generate to which languages and why
- which generator/tool is used per language and where manual adapters are required
- how exact decimals, date/time strings, enums, nullable fields, recursive refs, `oneOf`/`anyOf`, and additional properties are represented
- how source hashes, schema ids, and source file lineage are embedded or surfaced in generated packages

- how Swift/native coverage is decided:

  selected schema families consumed by the native workspace,
  or an explicit gap/coverage matrix
- how downstream runtime code is prevented from hand-editing generated outputs

Execution steps:
1. Start from the imported contracts package created in `pc_0060`.
   Treat it as canonical.
   Do not regenerate directly from ad hoc external files.
2. Create `binding_generation_matrix.json`.
   It must declare, per schema family:
   target languages,
   selected generator/tool,
   output path,
   naming rules,
   decimal strategy,
   optional/nullable strategy,
   and whether generation is expected to be lossless or gap-bearing.
3. Create `binding_naming_and_decimal_policy.json`.
   It must explicitly define:
   exact-decimal posture,
   enum casing/normalization,
   reserved-word handling,
   file/module naming,
   and when manual wrapper/adapters are required to preserve semantics.
4. Implement `generate_language_bindings.ts`.
   It must orchestrate deterministic generation, source-hash capture, output cleanup, and typed gap reporting.
5. Generate TypeScript bindings into `packages/generated-types/src/`.
   These must be ergonomic for the monorepo's browser/backend/shared packages and must remain clearly generated.
6. Generate Python models into the repo's approved generated-model location.
   Preserve compatibility with validator-adjacent tooling and avoid losing schema semantics through convenient-but-wrong coercions.
7. Generate Swift bindings for native-relevant schema families or, if full generation is not faithful,
   generate the maximal safe subset plus an explicit gap register describing what remains manual and why.
8. Create `binding_coverage_report.json`.
   It must summarize, by schema family and language:
   generated status,
   source hash,
   last generation time,
   and coverage class.
9. Create `binding_gap_register.json`.
   It must contain every schema feature or family the chosen generators could not faithfully express,
   with typed risk and required manual follow-on.
10. Build `binding-coverage-atlas` in the operator web internal route set.
    The page must let engineers inspect generation coverage, source hashes, tool choices, and gap posture without turning into a generic docs page.
11. Add unit/integration tests for generation determinism, coverage accounting, and source-hash lineage.
12. Ensure generated outputs are downstream only and clearly labeled as such.

Gap handling you must perform:
- Close the “the repo will soon need cross-language models, but there is no deterministic generation pipeline” gap.
- Close the “schema-to-language coverage is currently invisible, especially for Swift/native and Python tooling consumers” gap.
- Close the “exact-decimal and nullability semantics could drift during generation without an explicit policy” gap.
- Close the “downstream code could start hand-editing generated models without a clear source-hash and ownership boundary” gap.
- Close the “generator limitations would otherwise be hidden until runtime if there is no typed gap register” gap.

Mandatory edge cases:
- schemas using exact-decimal strings must not become lossy floating-point types by convenience
- recursive refs, unions, nullable combinations, and additional-properties semantics must either generate faithfully or enter the gap register
- Swift generation may not be able to represent the full schema corpus cleanly; report typed gaps rather than pretending full coverage
- Python and TypeScript generators may normalize enums or names differently; naming rules must remain deterministic and documented
- source-hash drift between the imported contracts package and generated outputs must be detectable
- generated files must remain clearly marked as generated and should not become the place where business logic accumulates
- compile/runtime smoke should remain scoped to representative families and not require every future app to exist before the generation pipeline is useful

Backend, performance, and security requirements:
- Keep generated outputs deterministic and reproducible.
- Keep the generation pipeline explicit, typed, and easy to rerun in CI.
- Preserve source-hash lineage in machine-readable output.
- Do not let generated models weaken runtime validation; schema validation remains canonical.
- Do not let the coverage atlas mutate generated code or source schemas.
- Prefer additive manual wrappers/adapters over editing generated files directly when a generator is insufficient.

Validation rules:
- Unit tests must prove every schema family in scope has a declared language-generation posture in the matrix.
- Unit tests must prove decimal/nullability naming policy is applied deterministically.
- Integration tests must prove generation can run repeatedly without spurious drift and that source hashes propagate into coverage reporting.
- Representative deserialization or compile smoke tests must exist for TypeScript, Python, and Swift outputs where generated.
- Playwright must validate semantic locators, keyboard flow, reduced-motion parity, inspector rendering, and lineage visibility on the atlas page.

Definition of done:
- A later implementation agent can consume generated TypeScript/Python/Swift contract models with confidence,
  know exactly which schema families are covered in each language,
  know where generator gaps remain,
  and continue building without hand-translating the contract corpus or losing source-of-truth lineage.
